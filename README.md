# Arkaim UI Registry

A shadcn-style registry plus a normative design contract for building consistent
admin panels. Extracted from the Arkaim admin frontend into this standalone repo so
it can be consumed via the shadcn CLI. See [`CHANGELOG.md`](./CHANGELOG.md) for
versions and [Publishing](#publishing) for how it reaches GitHub Pages.

This repo ships three things:

1. [`design-contract.md`](./design-contract.md) — the normative spec for what an
   admin panel built with this system looks like and how it behaves. **Read it
   first.**
2. [`registry.json`](./registry.json) + [`registry/`](./registry/) — the shadcn
   registry: tokens, ui-overrides, admin-primitives, app shell, hooks, lib.
3. [`skill/admin-ui/`](./skill/admin-ui/) — the Claude Code skill that bootstraps a
   project and scaffolds resource modules from the contract, plus a contract lint.

## How a consumer uses the registry

Pre-requisite: the consumer project has Tailwind v4, the shadcn CLI installed, and
a React + Vite or Next.js setup.

```bash
# 1. Install tokens (always first — everything depends on them).
npx shadcn add https://arkaim-labs.github.io/arkaim-ui-registry/r/tokens.json

# 2. Import the CSS from your global stylesheet, add the three Google Fonts.
#    (see tokens.css header and skill/admin-ui/bootstrap-project.md.)

# 3. Install what you compose with — registry deps come transitively.
npx shadcn add https://arkaim-labs.github.io/arkaim-ui-registry/r/app-sidebar.json
npx shadcn add https://arkaim-labs.github.io/arkaim-ui-registry/r/admin-page-layout.json
# … etc.
```

The shadcn CLI copies files into the consumer's project verbatim; the consumer owns
the code from that point. The full walkthrough is in
[`skill/admin-ui/bootstrap-project.md`](./skill/admin-ui/bootstrap-project.md).

## Validating locally

```bash
# structural checks (no install): files present + manifests match
node -e 'const r=require("./registry.json");console.log(r.items.length,"items")'

# contract lint over a consumer's source
node skill/admin-ui/lint/contract-lint.mjs <project>/src
```

`shadcn build` (which emits `public/r/*.json`) runs in CI on every push to `main`;
see the workflow in `.github/workflows/registry.yml`.

## Versioning

The registry follows the SemVer policy in section 16 of `design-contract.md`.
Releases are tagged `registry-v<X.Y.Z>`; the contract is tagged independently as
`contract-v<X.Y.Z>` and the skill as `skill-v<X.Y.Z>`. A pinned consumer points its
`npx shadcn add` URL at the tagged ref rather than `main`. See
[`CHANGELOG.md`](./CHANGELOG.md).

## Publishing

The repo publishes itself to GitHub Pages via `.github/workflows/registry.yml`
(runs `npx shadcn build` → uploads `public/`). No project install or lockfile is
needed — the build only reads `registry.json` and the source files. To bring it
online the first time:

1. Create the GitHub repo and push (needs a `gh` token with the `workflow` scope —
   `gh auth refresh -s workflow` if the push is rejected):
   `gh repo create Arkaim-Labs/arkaim-ui-registry --public --source . --remote origin --push`
2. Enable GitHub Pages: Settings → Pages → Source: **GitHub Actions**.
3. Trigger the workflow (push to `main`, or `gh workflow run "Build and publish registry"`).
   It publishes to `https://arkaim-labs.github.io/arkaim-ui-registry/`, items at
   `…/r/<name>.json`.
4. Tag the release:
   `git tag contract-v1.0.0 registry-v1.0.0 skill-v0.1.0 && git push --tags`,
   then flip the `unreleased` markers in `CHANGELOG.md` to dates.

Local build (optional): `npx shadcn@2.10.0 build` writes `public/r/*.json`.

The admin↔registry sync policy (backport generic changes both ways) lives in plan
38 of the originating monorepo.
